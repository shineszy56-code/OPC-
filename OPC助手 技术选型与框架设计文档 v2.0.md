## 1\. 项目概述

### 1\.1 项目背景：我们在解决什么问题

AI 让一个人能做过去需要整支团队才能完成的事。但工具体系没有跟上这个变化——一人公司创始人每天面对的，是一套为"团队协作"设计的工具世界，而不是为"一个人高效运转"设计的系统。

**问题一：工具极度碎片化，数据孤岛林立**
项目管理在 Notion 或 Linear，代码在 GitHub，个人笔记在 Obsidian，AI 对话在 Claude 或 GPT……这些工具之间没有任何连接。每次切换工具都是上下文的割裂，信息永远散落在不同系统里，没有一个地方能看到全貌。

**问题二：数据主权不在自己手里**
所有 SaaS 工具都把数据存在自己的服务器。商业计划、客户信息、产品方向全部托管在他人的数据库里——平台政策变化、账号封禁、服务关停，数据随时可能消失或被访问。

**问题三：临时协作的代价太高**
需要外包设计师看项目进度？要么截图，要么拉他注册账号。一人公司的协作需求是"偶发、临时、轻量"的，但现有工具为"长期、固定、重量"的团队设计，每次临时协作都要承担完整的注册、权限、账单成本。

**问题四：个人知识无法复利**
每个项目结束，经验随风散。AI 在执行任务时完全无法调用个人积累的知识——过去踩过的坑、形成的 SOP、读过的书、做过的决策，都只是躺在笔记软件里的死文档。

**问题五：讨论没有沉淀，想法消失在聊天记录里**
头脑风暴结束就散，没有人整理共识，没有人拆解行动项，最终只有一堆聊天记录无人翻阅。

**OPC助手** 就是为解决上述五个问题而生：一个本地优先、AI 原生、零账号的移动端全能助手。

### 1\.2 项目目标

打造一个**本地优先、AI 原生、零账号、轻量协作**的移动端全能助手 App，帮助一人公司创始人：
- 管理多个项目，自动同步 Git 开发进度
- 建立私人知识库，让个人智识资产持续复利
- 随时发起小组头脑风暴，AI 全程主持并归档成果
- 统一 AI 工具入口，实现"一个人就是一支军队"

### 1\.3 核心功能模块

#### 📋 项目与任务管理
- 零账号启动，所有数据本地存储
- AI 驱动的项目拆解、任务排期、进度追踪
- 自动生成日报、周报和项目报告
- 细粒度权限控制（只读/评论/编辑/管理）
- 成员操作日志记录，可追溯所有变更

#### 🔗 Git 同步引擎
- 根据 commit message / PR / Issue 自动更新任务状态
- 驱动项目进度实时计算，消除手动更新负担
- 自动同步 GitHub、GitLab、Gitea

#### 🤝 轻量协作
- 零注册临时链接分享（有效期最长 30 天）
- 全程走 Cloudflare 加密中继，端到端加密
- 实时同步多人编辑内容，CRDT 自动解决冲突

#### 🧠 知识管理（双层）
- **项目知识库（LLM Wiki）**：AI 持续摄入项目活动，自动整理知识页面，累积项目智慧
- **私人知识库（Personal KB）**：四层个人知识体系，全局跨项目，永远本地存储
  - A. 第二大脑：个人笔记/想法/读书摘录，AI 自动关联到相关项目
  - B. 文档库：导入 PDF、Markdown、TXT、网页 URL，AI 自动摘要索引
  - C. 工具同步：与 Obsidian、Notion、Logseq 双向同步，文件系统级集成
  - D. 能力库：个人 SOP 模板、Prompt 预设、工作流经验，可一键应用到项目

#### 💬 头脑风暴（Brainstorm）
- 小组实时协作讨论（2-10 人），零注册通过分享链接加入
- 实时消息 + 想法卡片，CRDT 自动同步，全程走 Cloudflare 加密中继
- 结构化讨论模式：自由发散 / SCAMPER / 六顶思考帽 / 玫瑰-荆棘-花蕾
- 会话结束后手动触发 AI 归纳总结，一键转化为任务、笔记或知识库页面
- ⚙️ AI 全程自动主持（自动追问/归纳/反驳）：规划中，暂不实现

## 2\. 核心设计原则

**所有技术决策必须严格遵循以下原则，不可动摇**

1. **绝对无状态云端**：云端不存储任何用户核心数据，仅作为临时数据交换通道

2. **数据主权 100%**：所有项目、任务、AI 生成内容永久存储在用户本地设备

3. **零账号零注册**：打开 App 即可使用，不需要任何个人信息

4. **完全离线可用**：所有核心功能在无网络环境下 100% 正常工作

5. **协作即临时**：所有协作数据都有有效期，到期自动永久删除

6. **去后端化**：主动探索弱后端/无后端架构范式，云端仅作无状态边缘中继，不承载任何业务逻辑和用户数据；Cloudflare 边缘网络是目前最贴近这一范式的基础设施选择

### 5\.4 知识库层设计（LLM Wiki 模式）

#### 5\.4\.1 设计理念

本模块基于 Andrej Karpathy 提出的 **LLM Wiki 范式**，将传统 RAG（每次查询重新检索原始文档）升级为**持久复利知识库**：AI 持续摄入项目活动，维护一套结构化的 Markdown 知识页面，知识随时间累积而非每次重新推导。

**与传统 RAG 的核心区别**：

|维度|传统 RAG|LLM Wiki（本项目）|
|---|---|---|
|知识存储|原始文档原样存储|AI 综合生成的结构化 Markdown|
|查询方式|向量相似度检索|索引引导 + AI 语义检索|
|知识积累|无状态，每次从零开始|持久复利，越用越聪明|
|矛盾处理|不处理|AI Lint 定期发现并解决|
|维护成本|人工维护文档|AI 自动维护，人类仅需提供方向|

#### 5\.4\.2 三层知识结构

#### 5\.4\.3 三种核心操作

**① Ingest（摄入）**：AI 读取新的项目活动，更新相关知识页面

**② Query（查询）**：AI 检索知识页面综合回答，有价值的洞见自动归档

**③ Lint（健康检查）**：定期检查知识库质量

#### 5\.4\.4 自动触发策略

|触发时机|操作|说明|
|---|---|---|
|任务状态变更|Ingest|将变更摘要写入相关实体页和 log.md|
|AI 工作流完成|Ingest|将 AI 输出的洞见归档为概念页或综合页|
|外部工具同步|Ingest|GitHub commit、Cursor 日志摘要写入知识库|
|用户主动提问|Query|回答后可选择归档洞见|
|每周定时|Lint|全库健康检查，结果推送本地通知|
|项目归档前|Lint + Ingest|生成完整项目复盘知识页面|

#### 5\.4\.5 隐私与安全

- 所有知识库文件存储在本地 App 沙盒目录，与 SQLite 数据库同级
- 知识页面内容参与本地 AES\-256\-GCM 加密备份
- 知识库**不通过 Cloudflare 同步**，协作场景下仅同步原始事件数据，各端独立维护自己的知识库
- AI 调用云端模型时，仅传输必要的上下文片段，不上传整个知识库

### 5\.7 Git 同步引擎设计

#### 5\.7\.1 设计目标

将 Git 仓库（GitHub / GitLab / Gitea）的实际开发进度**自动映射**到项目任务状态，消除手动更新进度的负担。提交、PR、Issue 的变化直接驱动任务状态流转和项目进度计算，同时将 Git 活动摘要注入知识库（LLM Wiki Ingest）。

#### 5\.7\.2 整体数据流

#### 5\.7\.3 数据模型

#### 5\.7\.4 commit message 解析规则

支持以下格式自动提取任务引用：

| 格式 | 示例 | 说明 |
|---|---|---|
| `#taskId` | `fix login bug #abc123` | 最简写法，关联任务 |
| `close #taskId` | `close #abc123` | 关联并自动将任务标记为 done |
| `ref #taskId` | `ref #abc123` | 仅关联，不改变任务状态 |
| 分支名前缀 | `feature/abc123-login` | 从分支名提取任务 ID |
| Conventional Commits | `feat(auth): add login` | 提取 scope 作为模块标签写入知识库 |

#### 5\.7\.5 状态映射规则

| Git 事件 | 条件 | 任务状态变更 | 项目进度影响 |
|---|---|---|---|
| commit（含 `close #id`）| 推送到主分支 | `→ done` | 重新计算 |
| commit（含 `#id`，无 close）| 任意分支 | `→ in_progress`（若当前为 todo）| 无 |
| PR opened（含 `#id`）| 任意 | `→ in_progress` | 无 |
| PR merged | 有关联任务 | `→ done` | 重新计算 |
| PR closed（未 merge）| 有关联任务 | 不自动变更，推送通知 | 无 |
| Issue closed | 有关联任务 | `→ done` | 重新计算 |

> 所有自动状态变更均写入 `OperationLog`，标注 `source: git_sync`，可追溯、可撤销。

#### 5\.7\.6 项目进度自动计算

#### 5\.7\.7 Cloudflare Webhook 中继 Worker

#### 5\.7\.8 API Token 安全存储

Git 平台的 Personal Access Token 使用 `flutter_secure_storage` 存储在系统 Keychain（iOS）/ Keystore（Android），**不写入 SQLite，不参与云端同步，不进入知识库**。

#### 5\.7\.9 与知识库（LLM Wiki）的集成

- **连续多次 commit 到同一模块** → AI 自动识别热点区域，写入综合页

## 5\.9 私人知识库层设计（Personal Knowledge Base）

### 5\.8\.1 设计定位

私人知识库是**跨越所有项目的个人智识资产**，与项目级 LLM Wiki 形成两层互补结构：

| 维度 | 项目知识库（LLM Wiki） | 私人知识库（Personal KB） |

| 隐私级别 | 可通过分享链接授权协作者访问 | 绝对私有，永不参与任何同步 |
| AI 调用 | 项目 AI 自动调用 | 由 AI 网关按需注入项目上下文 |

### 5\.8\.2 整体架构

## 13\. 去后端架构与传统后端架构的优劣势对比

### 13\.1 核心定义

- **本项目去后端架构**：所有核心数据存储、业务逻辑和计算都在用户端侧（手机/平板）执行，云端仅作为**临时、无状态的数据交换通道**，不存储任何用户核心数据，不执行任何业务逻辑。

- **传统后端架构**：所有核心数据存储在中心化服务器，业务逻辑和计算主要在服务器端执行，客户端仅负责UI展示和用户交互。

### 13\.2 去后端架构的核心优势

|维度|去后端架构|传统后端架构|本项目价值|
|---|---|---|---|
|**数据安全与隐私**|✅ 用户完全拥有数据主权，云端不存储任何明文数据，从根本上避免数据泄露和滥用风险|❌ 所有数据集中在服务器，存在被黑客攻击、内部人员泄露、第三方滥用的风险|极高：一人公司的项目数据包含商业机密、客户信息和AI生成内容，数据安全是第一优先级|
|**架构复杂度**|✅ 极低：云端无业务逻辑、无数据库、无状态，边缘节点仅做加密数据中继，Cloudflare 边缘网络天然支持这一范式|❌ 高：需要设计和维护服务器、数据库、API、缓存等完整后端体系|极高：一人公司没有专职后端工程师，弱后端化大幅降低系统复杂度和长期维护成本|
|**离线可用性**|✅ 所有核心功能100%离线可用，不受网络影响|❌ 几乎所有功能都需要网络连接，断网即瘫痪|高：创始人经常在移动场景下工作，需要随时查看和编辑项目|
|**性能**|✅ 数据和计算都在本地，无网络延迟，响应速度极快|❌ 所有操作都需要网络请求，受网络质量影响大|高：移动端操作对响应速度要求高，本地计算体验更好|
|**扩展性**|✅ 无限扩展：用户越多，计算资源越多（端侧提供），无需担心服务器扩容|❌ 扩展性有限：需要提前规划服务器容量，用户量突增可能导致服务崩溃|中：本项目用户量增长缓慢，但架构天然支持大规模用户|
|**运维复杂度**|✅ 极低：边缘侧仅有无状态 Worker 脚本，无数据库、无服务器、无备份压力；去后端化将运维复杂度降至接近零|❌ 运维复杂：需要专业的运维人员，负责服务器监控、备份、升级、安全等|极高：去后端化是降低系统长期维护成本的主动架构决策，而非对运维能力不足的被动妥协|
|**抗风险能力**|✅ 抗攻击、抗审查、抗中断：没有中心化服务器，单点故障不影响整体|❌ 单点故障风险高：服务器宕机导致所有用户无法使用|中：Cloudflare全球节点可靠性高，但仍存在服务中断风险|

### 13\.3 去后端架构的核心劣势

|维度|去后端架构|传统后端架构|本项目应对措施|
|---|---|---|---|
|**复杂业务逻辑实现**|❌ 所有逻辑都在端侧，难以实现需要全局状态或复杂计算的功能|✅ 服务器端可以轻松实现复杂的业务逻辑和全局计算|本项目业务逻辑简单，主要是项目和任务管理，不需要复杂的全局计算|
|**多人协作同步**|❌ 同步复杂：需要解决多人编辑冲突，CRDT算法有学习成本，同步延迟可能存在|✅ 同步简单：服务器作为唯一可信源，轻松实现多人协作|使用Yjs成熟的CRDT实现，自动解决冲突；限制单个项目最多10个协作者|
|**端侧性能限制**|❌ 移动端设备性能有限，无法运行复杂的AI模型或大数据处理|✅ 服务器可以使用高性能硬件，运行复杂的计算任务|优先使用量化本地模型处理简单任务，复杂任务交给云端AI服务|
|**数据备份与恢复**|❌ 依赖用户主动备份，设备损坏或丢失会导致数据永久丢失|✅ 服务器自动备份，数据恢复简单|提供本地文件备份和端到端加密云备份功能，提醒用户定期备份|
|**全局功能实现**|❌ 难以实现全局搜索、跨项目统计、用户行为分析等功能|✅ 服务器可以轻松实现全局数据统计和分析|本项目不需要全局功能，所有统计和分析都在本地完成|
|**应用更新**|❌ 需要用户主动更新App，无法无缝更新|✅ 后端更新对用户透明，不需要用户操作|使用 Shorebird 实现 Flutter App 的 OTA 静默热更新|
|**第三方集成**|❌ 很多第三方服务需要后端密钥，端侧存储密钥有安全风险|✅ 服务器可以安全存储第三方密钥，代理API请求|对于需要密钥的第三方服务，使用Cloudflare Workers作为代理|
|**数据一致性**|❌ 多设备同步复杂，可能出现数据不一致|✅ 服务器作为唯一可信源，保证数据一致性|以项目创建者的设备为唯一可信源，其他设备同步其数据|

### 13\.4 本项目的架构适配性结论

**去后端/弱后端架构是本项目的主动技术选择，而非成本妥协**，核心逻辑如下：

1. **架构哲学驱动**：本项目将"去后端化"本身作为一个值得探索的架构范式——验证端侧算力 + 边缘中继能否替代传统中心化后端，为 AI 时代的个人软件提供新路径。

2. **数据主权匹配**：去中心化架构天然实现数据主权，用户数据永不离开本地设备，无需依赖任何第三方保证——这是传统后端无法从根本上解决的信任问题。

3. **复杂度极简**：一人开发团队主动控制系统复杂度，弱后端化使整个服务端只需维护无状态 Cloudflare Worker 脚本，不存在数据库迁移、服务扩容、API 版本管理等传统后端问题。

4. **技术可行性已验证**：现代移动端设备性能充足；CRDT（Conflict-free Replicated Data Types）解决了多端同时编辑的冲突问题；Cloudflare KV 轮询提供简洁可靠的协作同步通道。

**边界说明**：当协作规模超过 10 人、需要全局数据聚合或复杂工作流引擎时，可渐进引入轻量级边缘函数承担特定计算，但核心原则始终是"本地优先，云端无状态"。
